## 1. 研究对象是什么？

我研究的是**多层仓库中的订单释放决策**——具体地说，是 WMS（仓库管理系统）在一个班次内把若干订单**如何分组、何时释放**给下游 AMR fleet 执行的决策。

这个看似"简单的上游决策"在真实多层仓库中有一个被长期忽视的结构性问题：**它对下游货梯的拥塞产生非线性影响**。两个看起来等价的订单分组方式（同样的订单数、同样的截止时间），在下游 AMR + 电梯系统中产生的 makespan 可以差 2–3 倍。

本研究的目标是把这个"**上游释放决策 → 下游电梯拥塞**"的耦合关系形式化，并给出一个可操作的优化框架。

---

## 2. 为什么这个问题值得研究？

有三个层面的动因，按从宏观到微观排列：

### 2.1 工业层面

多层仓库数量在快速增长（尤其东亚和欧洲都市圈），货梯已经成为多层 AMR 仓库的**首要吞吐瓶颈**。行业经验值：典型货梯吞吐 200–500 箱/小时，独立于 AMR 数量和软件优化。仓库要扩容，**买更多 AMR 不解决问题，建更多货梯成本极高**——唯一短期可行的路径是**优化 wave 释放决策**。

### 2.2 学术层面

三条独立的文献流在这里相遇，但无人把它们缝起来：

- **Wave planning 文献**：传统上假设单层 + 人工拣选，对 AMR 和电梯一无所知
- **AMR/MAPD 调度文献**：把 wave 视为外生给定，不优化组装
- **ASRS / VLM queueing 文献**：关注固定基础设施的性能分析，不涉及决策

**空缺位置明确**：没有工作把 wave composition 作为**显式决策变量**，与电梯容量约束联合优化。

### 2.3 方法论层面

这个问题天然是一个**两阶段决策问题**——Tactical 层决定 wave 组装和释放；Operational 层决定 fleet 执行。两阶段之间的**耦合非平凡**：一阶段决策完全改变二阶段的问题结构。这种结构在 PhD 博士论文的叙事中，恰好接续前两篇论文（都在纯 operational 层）——从而构成"**决策层级上升**"的自然递进。

## 3. 问题的物理场景是什么？

想象一个 4 层仓库。每层楼有若干拣选位置和一个电梯厅。楼层之间由 2–3 部货梯连接。仓库内有 20–50 台 AMR 执行订单运送。

一个典型的班次开始时：
- WMS 手中握有几百到上千个待处理订单
- 每个订单有明确的 source floor 和 destination floor
- 订单有 deadline 或优先级

WMS 必须决定：
- 这些订单如何分成若干个 "wave"（批次）
- 每个 wave 什么时候释放给 fleet

一旦 wave 被释放：
- Fleet scheduler 把 wave 内的订单分配给具体 AMR
- AMR 执行任务时，跨层的部分必须经过电梯
- 多台 AMR 同时争用电梯 → 拥塞发生

**关键物理约束**：电梯容量有限（一次只能载 1–3 台 AMR）、方向性的（上行和下行不能同时）、有切换成本（改变方向需要时间）。

> 🟡 数字可调：4 层、20–50 AMR、2–3 电梯是通用参考。改成你公司熟悉的具体仓库可以让叙事更有力。(我认为这个需要做鲁棒性分析)

---

## 4. 决策要回答什么？

本研究的决策分两个层次：

### 4.1 上层决策（Wave Composition & Release）

- 把 $N$ 个订单分成 $W$ 个 wave
- 每个 wave 什么时候释放

这两个决策是**一次性做出**的（在班次开始时），或者按**滚动视角**周期性更新。

### 4.2 下层决策（Fleet Execution）

- 每个订单分给哪台 AMR
- 每个跨层订单用哪部电梯
- 每台 AMR 的任务执行时序

这些决策**随时间动态做出**，响应 wave 的释放和电梯的当前状态。

### 4.3 两层的耦合点

上层决策直接影响下层可行域——**一个 wave 如果跨层任务集中在某一层，下层 fleet 无论怎么调度都会拥塞**。

这是本研究的**核心建模挑战**：如何让上层决策"看到"下层的物理约束？

## 5. 输入是什么？输出是什么？

### 5.1 输入

班次开始时已知：
- **订单池**：每个订单包含 source floor、destination floor、service time、deadline、priority
- **仓库配置**：楼层数、每层布局（抽象为平均层内移动时间）
- **Fleet 状态**：AMR 数量、初始位置、速度参数
- **电梯配置**：电梯数、容量、行程时间、开关门时间
- **规划周期长度**

### 5.2 输出

本研究的算法输出：
- **Wave 分配方案**：每个订单属于哪个 wave
- **Wave 释放时刻**：每个 wave 在什么时间点释放
- **（可选）Fleet execution 策略**：AMR-订单分配、电梯分配

### 5.3 评价指标

**主指标**（必须）：
- 总体 makespan（所有订单完成时刻的最大值）
- 或：加权完工时间（按订单优先级加权）
- 或：截止时间违约损失

**辅助指标**（用于 insight 分析）：
- 电梯利用率的均值和方差
- 跨层"空驶"比例
- Wave-to-wave 拥塞传递程度

## 6. 问题的核心难点是什么？

有四个难点，按严重程度排列：

### 难点 1：上下层的**递归反馈**

上层决策（wave 组装）依赖下层性能评估，但下层性能**取决于上层决策本身**。这是一个典型的**鸡生蛋**问题。通常的处理方式是：

- **精确方法**：把两层合并为一个大 MILP → 规模爆炸，通常不可行
- **近似方法**：用一个"**下层性能估计器**"$\Phi(\text{wave})$ 预测 makespan → 估计器精度直接决定上层决策质量
- **迭代方法**：上下层交替优化 → 收敛性不保证

v0.1 暂定用**近似方法**，估计器基于从仿真中学到的经验函数。

### 难点 2：**电梯的"隐式"建模**

电梯不是普通的加工机器——它是一个**带方向、带容量、带循环结构**的复杂服务资源。在 wave planning 的上层视角下，我们不能完整建模电梯的每一个开关门事件；需要用**等效吞吐模型**代替。

核心问题：什么样的 wave composition 特征能预测电梯的有效吞吐？初步假设是三维特征：
- 垂直集中度（tasks 在楼层上的分布）
- 方向性不平衡（上行 vs 下行比例）
- 时间集聚（wave 内任务的时间分布）

> 🟡 三维特征是推测。v0.2 可能会被数据证伪或补充。

### 难点 3：**真实仓库的不确定性**

实际运营中，订单服务时间、AMR 移动时间、电梯开关门时间都是随机的。v0.1 可以用确定性模型起步，但需要 v0.2 考虑 stochastic 版本。

### 难点 4：**scope 边界控制**

这个问题很容易滑向：
- 路径规划细节（这是 Paper 2 的 domain）
- 充电约束（这是 Paper 1 的 domain）
- 电梯内部调度的结构性理论（这是未来 BCCPS 的 domain）

保持 Paper 3 的 focus 在 **"wave composition → elevator congestion"** 这条链上，是整个研究的戒律。

## 7. 研究的科学假设是什么？

本研究围绕三个**可被反驳**的假设展开：

**H1**（存在性）：在真实工业 instance 上，vertical-aware 的 wave composition 相比 vertical-agnostic 的 baseline，至少能降低 15% 的 makespan。

**H2**（结构性）：vertical-aware 策略的收益**不均匀**——收益最大的场景是"高方向不平衡 + 紧电梯容量"的组合，收益最小的场景是"低不平衡 + 宽松容量"。

**H3**（可近似性）：一个基于三维特征的低阶代理模型 $\Phi$ 可以以 10% 内的误差预测 wave makespan，从而让上层决策可解。

**三个假设中任何一个被实验证伪，都是有学术价值的发现**——但三个都成立时，Paper 3 的价值最大。

> 🟡 15% / 10% 都是拍脑袋数字。有 intuition 告诉你应该是 10% 还是 30%，替换即可。

## 8. 解决方案的大致思路

本研究计划采用 **"建模 + 算法 + 验证"** 的三段式：

### 阶段 A：建模（Month 1–3）

- 用 MILP 形式化两阶段决策
- 同时建立 discrete-event simulation 作为"真实世界"的替身
- 确定 instance generator 的参数分布

### 阶段 B：算法设计（Month 4–7）

- **Baseline**：vertical-agnostic wave planning（经典 order batching 算法）
- **核心算法**：基于代理模型 $\Phi$ 的两阶段优化（ALNS 或 tabu search）
- **对比算法**：exact MILP（在小规模 instance 上）

### 阶段 C：验证（Month 8–10）

- 合成 instance 的 systematic study（验证 H2 的结构性）
- 真实数据的 case study（如果数据可得）
- 敏感性分析（验证方法的鲁棒性）

## 10. 我现在还不确定的问题

以下问题将在 Week 2 文献阅读和 Week 3 建模尝试中解决。每个问题对应一个具体的决策截止日期。

### Q1：Wave 的数量 W 是决策变量还是预先给定？

- 两种处理方式都有文献依据
- 倾向：v0.1 给定 W（例如 W = 3 或 W = 5），v0.2 再考虑变量化
- **决策截止**：Week 2 末

### Q2：电梯容量以什么单位度量？

- 选项 A：AMR 台数

### Q3：规划是**离线批量**还是**滚动视角**？

- 离线：规划期 T 开始时一次性决定所有 wave — 模型简单，但脱离真实运营
- 滚动：按固定频率（例如每 30 分钟）重新规划 — 更真实，但模型和算法都复杂
- 倾向：v0.1 先做离线，v0.2 扩展到滚动
- **决策截止**：Week 3 末

### Q4：下层性能估计器 Φ 用什么形式？

- 选项 A：解析公式（基于 queueing 近似）
- 选项 B：从仿真学习的 regression model
- 选项 C：蒙特卡洛抽样
- 倾向：v0.1 探索所有三种，比较精度-速度权衡
- **决策截止**：Month 2

### Q5：Single-elevator 还是 Multi-elevator 作为起点？

- 倾向：v0.1 先 E = 1（最小可解），v0.2 扩展到 E ≥ 2
- **决策截止**：v0.1 不改

### Q6：Wave 内的订单释放是**瞬时**还是**可 staggered**？

- 瞬时：wave 释放时所有订单同时进 fleet queue
- Staggered：wave 内订单按某个 schedule 陆续释放
- 后者可能是一个 interesting sub-contribution
- **决策截止**：v0.2 讨论，v0.1 先用瞬时
